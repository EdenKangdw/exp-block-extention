# 개발 시 고려사항 (Considerations)

1. **데이터 무결성 및 보안**
   - **입력 검증**: 커스텀 확장자 추가 시 길이(20자) 및 형식(영문/숫자)을 엄격히 제한하여 잘못된 데이터나 스크립트 주입을 방지했습니다.
   - **중복 방지**: 이미 등록된 확장자는 중복 등록되지 않도록 처리했습니다.

2. **사용자 경험 (UX)**
   - **실시간 피드백**: 잘못된 입력이나 중복 시 즉각적인 알림(Alert)을 제공합니다.
   - **시각화**: 커스텀 확장자를 태그 형태로 보여주어 직관적으로 관리할 수 있게 했습니다.
   - **편의성**: 개별 삭제뿐만 아니라 '전체 삭제' 기능을 추가하여 테스트 및 초기화 편의성을 높였습니다.

3. **배포 및 확장성**
   - **환경 변수 지원**: `PORT` 환경 변수를 지원하여 Cloudtype 등 다양한 배포 환경에서 유연하게 동작하도록 했습니다.
   - **경량화**: FastAPI와 SQLite를 사용하여 가볍고 빠르게 동작하며, 별도의 복잡한 DB 설정 없이 바로 실행 가능합니다.

4. **데이터베이스 설계 (Database Design)**
   - **단일 테이블 통합**: 초기에는 고정/커스텀 확장자를 별도 테이블로 관리했으나, 확장성과 관리 효율성을 위해 `file_extensions` 단일 테이블로 통합했습니다.
   - **유연성 확보**: `type` 컬럼을 통해 확장자 유형을 구분함으로써, 추후 새로운 유형(예: 관리자 차단, 임시 허용 등)이 추가되더라도 스키마 변경 없이 대응 가능합니다.
   - **일관성 유지**: 모든 확장자에 대해 허용 여부(`is_allowed`), 수정자(`update_by`), 수정일시(`update_at`)를 일관되게 관리하여 데이터 이력 추적이 용이합니다.

5. **대소문자 처리 (Case Sensitivity)**
   - **소문자 정규화**: 파일 확장자는 대소문자를 구분하지 않는 것이 일반적이므로, 모든 입력값은 저장 시 **소문자로 변환**하여 저장합니다.
   - **중복 방지**: 이를 통해 `Test`, `TEST`, `test` 등이 서로 다른 확장자로 인식되어 중복 등록되는 문제를 방지했습니다.
